# Glossary

| Term                       | Meaning                                                                                                                                                             | Use in code/specifications                    | Avoid                            |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------- | -------------------------------- |
| Ambiguous Outcome          | A submission outcome for which the client cannot determine whether the external service processed the request.                                                      | `ambiguous outcome`, `AMBIGUOUS`              | unknown error, maybe failed      |
| Batch                      | A multi-entry CLI operation that accepts multiple Job Entries and associates the resulting Jobs for aggregate reporting while retaining independent Job state.       | `batch`, `batch_id`                           | bulk request, job group          |
| Duplicate Submission       | An additional submission of a Job that has already been accepted or may already have been processed. A matching `SUCCEEDED` Job is returned as `already_completed` without another external request. | `duplicate submission`, `already_completed` | duplicate job, repeated click |
| Execution State            | Persisted information describing the current lifecycle stage and known outcome of a job.                                                                            | `execution state`, `state`                    | status data, cache               |
| External Service           | The HTTP service to which the CLI submits jobs. In this project, it is represented by an intentionally unreliable simulator.                                        | `external service`, `job API`                 | backend, third party             |
| Idempotency Key            | A stable identifier assigned to a logical submission and used to associate its repeated delivery attempts. The application workflow generates one by default, or the CLI accepts one from the user; it is never derived from the Job Entry payload. | `idempotency key`, `idempotency_key` | request ID, retry ID |
| Job                        | The authoritative local entity representing one unit of work, its current execution state, and its attempt history.                                                | `job`, `job_id`                               | task, request                    |
| Job Entry                  | A user-provided JSON object describing one unit of work to be submitted to the external service.                                                                    | `job entry`, `job_entry`                      | job request, task                |
| Permanent Failure          | A failure that must not be retried automatically because another attempt is not expected to succeed without changing the input or configuration. A SubmissionAttempt records `PERMANENT_FAILURE`; the resulting terminal Job state is `FAILED_PERMANENT`. | `permanent failure`, `PERMANENT_FAILURE`, `FAILED_PERMANENT` | fatal error, hard fail |
| Rate Limit                 | A restriction imposed by the external service on how frequently requests may be submitted.                                                                          | `rate limit`, `rate_limited`                  | throttle error, API busy         |
| Reconciliation             | The process of resolving an ambiguous outcome by checking external evidence or requiring explicit human action.                                                     | `reconciliation`, `reconcile`                 | retry, recovery                  |
| Remote Request ID          | An identifier returned or exposed by the external service for a request. It may be absent or duplicated and does not by itself prove a unique or definitive remote outcome. | `remote request ID`, `remote_request_id` | job ID, primary key              |
| Retry                      | A subsequent delivery attempt for the same logical submission, preserving its identity and idempotency key.                                                         | `retry`, `retry attempt`                      | resubmit, send again             |
| Retryable Failure          | The `RETRYABLE_FAILURE` outcome of a SubmissionAttempt for which another attempt may reasonably succeed. A Job awaiting that attempt is `RETRY_SCHEDULED`.            | `retryable failure`, `RETRYABLE_FAILURE`      | temporary error, transient issue |
| Simulated External Service | The local HTTP server that provides idempotent submission and reconciliation by idempotency key while reproducing defined unreliable behavior for both operations.    | `simulated external service`, `simulator`     | mock server, fake backend        |
| Structured Log             | A machine-readable JSON record describing an operational event and its associated context.                                                                          | `structured log`, `log event`                 | debug output, console message    |
| Submission                 | The logical act of delivering one job entry to the external service. A submission may consist of multiple attempts.                                                 | `submission`                                  | request, execution               |
| Submission Attempt         | One recorded delivery attempt that may result in one HTTP interaction with the external service.                                                                    | `submission attempt`, `attempt`, `attempt_id` | retry request, call              |
