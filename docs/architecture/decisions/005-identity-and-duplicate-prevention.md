# ADR-005: Identity and Duplicate Prevention

## Status

Accepted

## Date

2026-07-16

## Context

`boolean-maybe` needs a stable local identity for each Job and a stable delivery identity that both the application and simulated external service can use to recognize one logical submission. These identities serve different boundaries and must not be conflated with each other, the Job Entry payload, or an unreliable remote request ID.

The product already permits either an application-generated or user-provided idempotency key, forbids deriving that key from the payload, and requires a matching completed Job to be returned without another external request. ADR-003 defines simulator-side idempotency and RFC 8785 Job Entry equivalence. ADR-004 defines the transaction and uniqueness boundary that must serialize duplicate-prevention decisions across CLI processes.

This ADR defines the application-side identity model and connects those accepted contracts without changing the Job lifecycle or defining the detailed CLI surface.

## Decision

### Separate identity roles

`job_id` and `idempotency_key` are both immutable, but they have distinct roles:

| Identity | Authority and role | Boundary |
| --- | --- | --- |
| `job_id` | Locally generated primary identity of one persisted Job. It addresses that Job and its attempt history regardless of remote evidence. | Local application and persistence only; it is not the simulator's idempotency identity. |
| `idempotency_key` | Stable identity of one logical submission and all of its delivery attempts. It binds exactly one local Job to one equivalent Job Entry. | Persisted locally and transmitted unchanged to the simulator for submission and reconciliation. |
| `remote_request_id` | Optional remote evidence associated with processing. | Never local identity, never a uniqueness key, and never a substitute for either identity above. |

The application generates `job_id` independently when it creates a Job. The value is opaque and collision-resistant; its exact format is an implementation detail selected by the applicable persistence feature specification. If a generated `job_id` collides with an existing local identity, the application generates another value and retries creation; a `job_id` collision has no Job-reuse semantics.

### Idempotency-key origin and ownership

The application workflow resolves the idempotency key before Job creation:

* when the user supplies a key, the application validates and preserves that opaque value;
* otherwise, the application generates a new opaque, collision-resistant key;
* generated-key collision with an existing local binding must cause generation of another key before Job creation, not reuse of the existing Job;
* validation must reject values that cannot be transported through the ADR-003 HTTP contract;
* exact key format, length limits, allowed characters, and generation encoding are defined by an approved feature specification and must be compatible across the CLI, persistence adapter, HTTP client, and simulator.

The application must not case-fold, trim, decode, or otherwise normalize two distinct accepted key values into one identity. Transport encoding, including URI encoding for reconciliation, does not change the underlying key. The feature-level validation and encoding contract must define collision-free, reversible mappings between every accepted key value and both wire representations: the submission header value and reconciliation path value. Two distinct accepted keys must never produce the same wire identity, and each received wire value must decode to exactly one accepted key.

The key is generated by the application workflow rather than the CLI adapter. The CLI adapter may accept a user-provided value and present the resolved value, but it does not own identity or duplicate-prevention rules.

### Why the key is not payload-derived

An idempotency key must never be generated deterministically from the Job Entry or its digest.

Equivalent payloads can represent separate intentional submissions and therefore may require different logical identities. Conversely, a user-provided key intentionally links repeated invocations to the same logical submission. Payload-derived keys would collapse those cases, make identity depend on content, and make a payload digest act as an authority it is explicitly not intended to be.

RFC 8785 canonical bytes and their optional SHA-256 digest are used only to compare the payload bound to an existing key. They do not create or replace `job_id` or `idempotency_key`.

### Job Entry equivalence and key binding

One accepted `idempotency_key` binds to exactly one complete Job Entry according to the RFC 8785 canonical-byte equality defined by ADR-003 and `docs/domain/core-entities.md`.

The application retains the provenance of the resolved key until Job creation commits. A key supplied by the user, or a persisted key reused for an existing Job, carries intentional cross-invocation identity. A key newly generated for the current invocation and not yet persisted does not.

Within the ADR-004 pre-side-effect transaction, the application applies these rules to an intentionally reused key:

1. If no local Job is bound to the key, create one Job with a new `job_id`, the resolved key, and the immutable Job Entry.
2. If a Job is already bound to the key and its Job Entry is equivalent, reuse that Job; do not create another Job.
3. If a Job is already bound to the key and its Job Entry is non-equivalent, reject the request as an idempotency conflict before creating a SubmissionAttempt or initiating HTTP.

For a key newly generated in the current invocation, any collision with an existing binding—whether discovered before insert or after losing a concurrent uniqueness race—is an internal generation collision regardless of Job Entry equivalence. The application must generate another key and repeat Job creation; it must not reuse the winning Job or report an idempotency conflict. Once the generated key is committed on a Job, later uses of that persisted key follow the intentional-reuse rules above.

The persistence uniqueness constraint on `Job.idempotency_key` is the final local race boundary. A process that loses a concurrent create race for an intentionally reused key must re-read the winning binding and apply the equivalence rules; it must not create another Job or send from stale state.

### Matching completed Job and stored-result replay

For this contract, a matching completed Job is an existing `SUCCEEDED` Job with the same idempotency key and an equivalent Job Entry.

The application returns the authoritative persisted successful result with:

* `outcome = "already_completed"`;
* `submitted = false`;
* process exit code `0`.

`submitted = false` means that the current CLI invocation did not initiate another external request. The application creates neither a new Job nor a new SubmissionAttempt for this replay.

The replay is materialized from the existing authoritative Job and its persisted successful attempt evidence. It is not a fresh simulator response and does not require a reconciliation call. Presentation schemas and which diagnostic fields are repeated are defined by the applicable CLI feature specification.

`already_completed` does not define the behavior of an equivalent existing Job in `PENDING`, `SUBMITTING`, `RETRY_SCHEDULED`, `FAILED_PERMANENT`, or `AMBIGUOUS`. Applicable submission, retry, reconciliation, and recovery specifications must define their presentation while preserving the existing lifecycle and the rule that duplicate detection alone cannot authorize another request.

### Simulator propagation and remote conflict

Every SubmissionAttempt for one Job sends the Job's exact persisted `idempotency_key` in the `Idempotency-Key` header required by ADR-003. Retry never creates a new key. Reconciliation uses the same key as the URI-encoded path value and does not submit the Job Entry.

The local `job_id` is not used as the simulator idempotency key and need not be exposed as protocol identity. Remote request IDs returned by the simulator remain evidence only.

The simulator independently enforces the same key-to-equivalent-payload binding under ADR-003:

* an equivalent replay returns its stored processed result without processing again;
* non-equivalent reuse returns `409 Conflict` without changing the stored binding or processing the conflicting payload.

A delivered simulator `409` is authoritative evidence of a remote key conflict. The application must not replace the local Job Entry, adopt the simulator's conflicting payload, generate a replacement key silently, or report the local Job as successful. Its lifecycle classification follows the permanent-failure and recovery rules already defined by the domain model and ADR-004.

## Consequences

Benefits:

* Local addressing, logical submission identity, payload comparison, and remote evidence have explicit non-overlapping roles.
* User-provided keys support deliberate replay across invocations, while generated keys make independent submissions distinct by default.
* Equivalent duplicate requests reuse one authoritative local Job, including safe local replay after success.
* Non-equivalent key reuse is rejected before a local side effect and independently protected at the simulator boundary.
* Retry and reconciliation preserve one end-to-end logical identity without trusting remote request IDs.

Trade-offs:

* Users who want a later invocation to resolve the same logical submission must retain or deliberately provide its idempotency key.
* Identical payloads with independently generated keys are intentionally distinct Jobs and may be processed separately.
* Both application and simulator must implement the same canonical equivalence rule.
* Key validation must satisfy HTTP header, URI, persistence, and usability constraints without implicit normalization.
* Duplicate prevention reduces accidental resubmission but does not provide exactly-once remote processing when submission and reconciliation are unreliable.

## Impact on Core Entities

None.

This decision preserves the existing `Job` fields and invariants: `job_id` remains the unique local identity, `idempotency_key` remains a unique immutable logical-submission identity, and `payload` remains immutable. It does not add a digest or stored-result field to `Job` or `SubmissionAttempt`, change a lifecycle state, or alter entity ownership.

No update to `docs/domain/core-entities.md` is required for this decision. The future identity/submission feature specification must demonstrate that the existing persisted Job and successful SubmissionAttempt evidence can materialize its concrete `already_completed` response. If the approved response contract requires additional authoritative persisted data, that change requires explicit approval and synchronization of the affected core-entity contract before implementation; this ADR does not authorize adding it silently.

## Alternatives Considered

1. **Use `job_id` as the idempotency key.**
   Rejected because local entity addressing and cross-invocation logical submission identity have different roles. It would also prevent a user from supplying a stable key before a local Job exists.

2. **Derive the idempotency key from canonical Job Entry bytes or their digest.**
   Rejected because identical content may represent distinct intentional submissions, while identity must remain a deliberate choice independent of payload content.

3. **Use the remote request ID as local identity.**
   Rejected because it may be absent or duplicated and exists only after some remote processing paths.

4. **Treat any reuse of a key as a successful duplicate.**
   Rejected because a non-equivalent payload could silently receive an unrelated stored result.

5. **Create a new local Job for every invocation and deduplicate only at the simulator.**
   Rejected because local history and lifecycle authority would fragment, concurrent processes could initiate redundant HTTP requests, and stored-result replay would depend unnecessarily on an unreliable service.

6. **Generate a replacement key automatically after a local or remote conflict.**
   Rejected for a user-provided or already persisted key because it would silently turn an intentional-identity error into a new logical submission and could create an unintended remote side effect. This does not apply to collision recovery for a newly generated, not-yet-persisted key, which has not established logical identity and must be regenerated as defined above.

7. **Call reconciliation before returning every matching successful Job.**
   Rejected because the local completed result is already authoritative for application behavior and reconciliation is itself unreliable.

## Scope

This decision applies to:

* the distinct roles of `job_id`, `idempotency_key`, payload equivalence, and remote request IDs;
* user-provided and application-generated idempotency keys;
* local key binding and duplicate-prevention decisions;
* matching successful Job replay without another attempt or HTTP request;
* propagation of one key through submission, retry, and reconciliation;
* local and remote handling of non-equivalent key reuse.

This decision does not select or define:

* concrete identifier formats or generation libraries;
* exact key length, character set, validation messages, or CLI option names;
* physical persistence schema or SQL;
* complete CLI request and response schemas;
* behavior and presentation for every non-`SUCCEEDED` existing Job state;
* retry, reconciliation, recovery, retention, or simulator-reset policy;
* authentication, authorization, or secret-management policy.

## Follow-up on Acceptance

If this ADR is accepted:

* update `docs/architecture/architecture-overview.md` to reference the decided separation between local Job identity and logical submission identity;
* add this ADR to the overview's Related Architecture Decisions section;
* clarify in `docs/product/product-brief.md` and `docs/product/glossary.md` that “matching completed Job” in the `already_completed` contract means a matching `SUCCEEDED` Job;
* require applicable feature specifications to define key validation and generation format, persisted result materialization, non-success existing-state presentation, and race tests;
* require contract tests for generated and user-provided keys, generated-key and `job_id` collisions before and during concurrent creation, equivalent replay, non-equivalent local conflict, concurrent intentional same-key creation, transport round trips and distinct-key wire identity, matching `SUCCEEDED` replay, stable retry propagation, and simulator `409` handling.

No core-entity synchronization is required by this decision. The product brief and glossary need only the terminology clarification described above; their behavior remains unchanged.
